## 學習目標

完成本單元後，您將能夠：

## 內容大綱

當 Dify 內建的工具以及基於 OpenAPI 的「自定義工具」仍無法滿足您複雜的業務需求時，Dify 提供了最強大的自定義機制：**擴展插件 (API Based Extension)**。

透過擴展插件，您可以自己撰寫一段代碼（或者一整個微服務），並將其註冊到 Dify 中，讓 AI 能夠像呼叫一般工具一樣，呼叫您自製的服務。

### 1. 什麼是 API 擴展插件？

API 擴展插件本質上是一個 **由您自己託管的 Web 服務 (Web Service)**。
當 Dify 的智能體 (Agent) 或工作流 (Workflow) 需要執行這項工具時，Dify 會向您的伺服器發送一個包含參數的 HTTP POST 請求。您的伺服器收到請求後，執行相對應的程式邏輯，最後把結果包裹成 Dify 要求的 JSON 格式並回傳。

**與「自定義工具 (OpenAPI)」的差異**：
*   **自定義工具 (OpenAPI)**：服務端已經有一個標準的 RESTful API，您只需要把 Swagger/OpenAPI 文件貼給 Dify，Dify 就知道怎麼打這個 API。**程式邏輯全在原有系統中**。
*   **API 擴展插件**：您擁有極致的自由度，可以在自己的伺服器上撰寫任意的腳本 (例如：連接私有資料庫爬梳資料、控制本地端的硬體設備、呼叫另一個不對外公開的內網 API)。只要您的 Web 服務介面符合 Dify 的規範即可。

### 2. API 擴展的基礎規範

為了讓 Dify 能夠成功與您的 Web 服務溝通，您的服務必須實作特定的端點 (Endpoints) 與驗證機制：

1.  **驗證機制 (Authentication)**：
    Dify 在呼叫您的服務時，會在 HTTP Header 帶上 `Authorization: Bearer <您的_API_KEY>`。您的服務必須驗證這個 Key，確保請求是真的來自您的 Dify 應用。

2.  **核心端點 (Endpoints)**：
    通常您的服務需要提供一個 `/invoke` (名稱可自訂) 的 POST 端點。
    *   **輸入 (Request)**：Dify 會把 AI 決定好的參數 (例如：`{"user_query": "查詢昨日營收"}`) 透過 JSON Body 傳送過來。
    *   **處理 (Process)**：您的系統收到參數後，連接關聯式資料庫 (SQL) 撈取資料並進行計算。
    *   **輸出 (Response)**：您的系統將整理好的結果，或者是純文字，按照 Dify 規範的 JSON 格式回傳。

### 3. 擴展插件的實務開發場景

開發一個專屬的 API 擴展插件，可以讓您的 Dify 應用具備以下進階場景：

1.  **高度隱私的資料庫查詢**：不希望將資料庫開放到公網，就在內網寫一個 API Extension。AI 想查資料時，呼叫這個 Extension，Extension 查完再把結果告訴 AI。
2.  **執行複雜腳本**：有些邏輯非常複雜，無法用語意輕易描述，或是 Dify 的代碼節點效能不夠（如：對大型圖片進行浮水印去除、執行耗時的機器學習推理）。將其封裝成 Extension，讓 AI 負責「何時呼叫」，Extension 負責「苦力活」。
3.  **串接舊有系統 (Legacy Systems)**：舊公司系統使用的是 SOAP 或甚至沒有 API 而是靠 RPA 工具？您可以寫一個中介伺服器作為 Dify 跟舊系統之間的橋樑。

### 4. 總結

API 擴展插件是 Dify 向外延伸最核心的機制。只要您會寫基礎的 Web 程式 (Python FastAPI, Node.js Express, Go 等)，就能賦予 AI 無限的可能性，真正實現「AI 驅動的自動化流程」。
